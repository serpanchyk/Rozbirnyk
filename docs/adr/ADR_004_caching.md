# ADR 004: Resilient Decorator-Based Caching with Redis

## Status
Accepted

## Context
In the Rozbirnyk multi-agent system, agents frequently request the same external data (e.g., specific news events via Tavily) during a single simulation run. Hitting external APIs repeatedly for identical queries introduces severe latency, rapidly depletes API credits, and risks logical inconsistencies if the external data changes mid-simulation. 

We need a centralized caching mechanism. However, if the cache infrastructure goes down, it should not cause a hard crash of the entire simulation engine.

## Decision
We will implement a resilient, Redis-backed caching layer in the `common` package, exposed primarily through a `@cache_tool` asynchronous decorator.

### 1. Technical Implementation
*   **Decorator Pattern:** Caching is applied declaratively via `@cache_tool(namespace="...", ttl_seconds=...)` rather than hardcoding Redis logic inside business functions.
*   **Deterministic Key Generation:** Cache keys are generated by taking an MD5 hash of a sorted, JSON-serialized representation of the function's `args` and `kwargs`. This ensures identical inputs always hit the same cache key.
*   **Singleton Connection:** `get_redis()` ensures only one connection pool is created per application instance, configured via Pydantic `CacheSettings`.

### 2. Graceful Degradation
The caching layer wraps Redis `GET` and `SET` operations in `try...except ConnectionError` blocks. If the Redis container (`rozbirnyk-redis`) is unavailable or crashes:
*   The system logs a warning/error.
*   The cache layer explicitly bypasses Redis.
*   The underlying tool executes normally.

### 3. Observability Integration
The cache module utilizes the `common.logging` standard to emit structured JSON logs for `Cache HIT`, `Cache MISS`, and connection failures, injecting the `namespace` to track which specific tools are generating the most hits.

## Consequences

### Positive
*   **Performance & Cost:** Drastically reduces latency and API spend for repeated tool calls across ReAct loops.
*   **High Availability:** The graceful degradation ensures the simulation engine remains operational even if the caching infrastructure fails.
*   **Clean Code:** The decorator pattern abstracts caching logic completely out of the MCP tool implementations.

### Negative
*   **Serialization Limits:** The MD5 hashing relies on `json.dumps`. Tools decorated with `@cache_tool` cannot accept non-serializable arguments (e.g., complex class instances, raw bytes).
*   **Stale Data:** Hardcoded TTLs mean data might become outdated during long-running sessions unless explicitly flushed.

### Mitigation
*   Tool developers must ensure input parameters are primitive, JSON-serializable types (strings, ints, lists, dicts).
*   Redis health checks are defined in Docker Compose to ensure the container automatically restarts if it enters a bad state.
